# release

Open a release-bump PR for kura. The skill walks the commits since the
last release tag, classifies them, computes the next semver, and opens a
PR that bumps the version and promotes the changelog. Tagging and
publishing happen later — this skill stops at the PR.

## Mental model

`.github/workflows/release.yml` triggers on a pushed tag matching `v*`.
So the release flow is:

1. **(this skill)** Open `chore(release): bump to vX.Y.Z` on a
   `release/vX.Y.Z` branch.
2. Maintainer reviews + merges.
3. Maintainer tags the merge commit:
   `git tag vX.Y.Z && git push origin vX.Y.Z`. `release.yml`
   cross-compiles the five targets, creates the GitHub Release with the
   archives + `SHA256SUMS.txt`, and (when `HOMEBREW_TAP_TOKEN` is set)
   generates and pushes `Formula/kura.rb` to `kechol/homebrew-tap`.
   `docs.yml` publishes the docs site for the same tag.

Steps 2–3 stay manual. The skill never tags, never pushes to `main`,
never publishes.

## Pre-flight

Refuse to start when any of these fail, and stop on the first failure:

1. **Clean worktree.** `git status --porcelain` is empty.
2. **On `main`, in sync with origin.** `git fetch origin`, then both
   `git rev-list --count origin/main..HEAD` and
   `git rev-list --count HEAD..origin/main` are 0.
3. **`gh` authenticated.** `gh auth status`.
4. **At least one commit since the last tag.** Otherwise nothing to
   release.
5. **No existing `release/v*` branch on the remote.**
   `gh api repos/{owner}/{repo}/branches | jq -r '.[].name' | grep '^release/'`
   returns nothing. Manual cleanup beats a half-merged bump.

## Picking the next version

1. **Current version** — `version` in `package.json` (source of truth).
2. **Last tag** — `git describe --tags --match 'v*' --abbrev=0`.
3. **Commits since** — `git log <tag>..HEAD --format='%H%x09%s%x09%b%x00'`
   (NUL-terminate; commit bodies contain newlines).
4. **Classify each commit** (Conventional Commits):
   - body has `BREAKING CHANGE:` OR subject has `!` after the type
     → **breaking**
   - subject starts with `feat[(scope)]:` → **feature**
   - subject starts with `fix[(scope)]:` → **fix**
   - everything else (`chore`, `ci`, `docs`, `refactor`, `test`,
     `build`, `perf`) → noted, doesn't drive the bump
5. **Highest-wins bump**: any breaking → **major**; else any feature →
   **minor**; else any fix → **patch**; else (only chore/docs/etc.) →
   **patch**, but flag in the summary that this is a docs/build-only
   release and let the user override.
6. **Pre-1.0 convention** (`major == 0`): collapse "breaking" and
   "feature" both to **minor**. Patch stays patch. Keep this transparent
   in the printed plan so the user can override.

Print the plan before any write:

```
Current: v0.1.0
Last tag: v0.1.0  (HEAD: 21d7f0a)
Commits since: 18 total
  feat:        4
  fix:         2
  breaking:    1   (pre-1.0 → counted as minor)
  chore/docs:  11
Proposed: v0.2.0 (minor)
Branch:   release/v0.2.0
```

Ask the user to confirm or override the version. Validate any override is
strictly greater than the current.

## Files to bump

- `package.json` — `"version": "X.Y.Z"` (no leading `v`).
- `bun.lock` — refresh with `bun install` (kura's own version can appear
  in the lockfile). If it's a no-op, that's fine; commit only if it
  actually changed.
- `CHANGELOG.md` — see the next section. Mandatory.

Do NOT touch:

- `docs/package.json` — the docs site is independently versioned
  (`0.0.0`, private).
- `README.md` / `README.ja.md` / `docs/` prose — unless the user asks;
  version strings there are advisory.
- The Homebrew formula — generated by `release.yml` from the built
  archives' checksums, not hand-edited here.

## Updating `CHANGELOG.md`

kura's `CHANGELOG.md` follows [Keep a Changelog](https://keepachangelog.com/)
with an `## [Unreleased]` section and `### Added` / `### Changed` /
`### Fixed` / `### Removed` sub-sections. The flow is **promote +
categorize + reset**:

1. Rename `## [Unreleased]` to `## [X.Y.Z] - YYYY-MM-DD` (today, UTC).
2. Inside the renamed section, keep the existing `### Added` / `###
   Changed` / `### Fixed` / `### Removed` buckets. Ensure every entry is
   in the right bucket. Drop empty buckets.
3. **Cross-check git log**: `git log <last-tag>..HEAD --format='%h %s'`.
   For each Conventional Commit subject — `feat(...)` → Added (or a
   `Changed`/breaking note if `!`), `fix(...)` → Fixed, removals →
   Removed. Any user-visible commit missing from the section gets a
   one-line bullet: `- <description> (\`<short-sha>\`)`. `chore` / `test`
   / plain `refactor` / plain `docs` are optional — include only if they
   shipped a user-visible improvement.
4. Re-add a fresh empty `## [Unreleased]` heading above the new versioned
   section so future PRs have a landing zone:

   ```markdown
   # Changelog

   All notable changes to this project will be documented in this file.
   The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

   ## [Unreleased]

   ## [X.Y.Z] - YYYY-MM-DD

   ### Added
   - ...
   ```

5. Style: plain English, ≤ 80-char lines, lead each bullet with **what
   changed for the user** not the internal mechanism; for a breaking
   change add a short "Migration:" note showing before / after.

## Branch + commit shape

```sh
git switch -c release/vX.Y.Z
# ...bump package.json + bun install + CHANGELOG promote...
git add package.json bun.lock CHANGELOG.md
git commit -m "chore(release): bump to vX.Y.Z"
```

Commit body: a categorized summary, most relevant first:

```
chore(release): bump to v0.2.0

Highlights since v0.1.0:

Added
- feat(search): query expansion via local LLM (64a848c)
- ...

Fixed
- fix(db): trigram fallback when vaporetto dlopen fails (abc1234)

Breaking
- (none — pre-1.0 minor bump)
```

If a `BREAKING CHANGE:` trailer exists, quote it verbatim.

## Sanity checks before pushing

Run locally before `git push`; if anything fails, leave the branch local
and surface the error:

- `bun install --frozen-lockfile` (confirms the lockfile is settled)
- `bun run check` (`tsc --noEmit` + `biome check`)
- `bun test`

CI re-runs these, but a green local run avoids a red CI on the release PR.

## Opening the PR

```sh
git push -u origin release/vX.Y.Z
gh pr create \
  --base main \
  --head release/vX.Y.Z \
  --title "chore(release): bump to vX.Y.Z" \
  --body "$(cat <<'EOF'
## Summary
- Bumps kura from vA.B.C → vX.Y.Z
- CHANGELOG.md "[Unreleased]" promoted to the vX.Y.Z section

## Changelog

<paste the new [X.Y.Z] section from CHANGELOG.md verbatim>

## Release checklist (after merge)
- [ ] Tag the merge commit: `git tag vX.Y.Z && git push origin vX.Y.Z`
- [ ] Confirm `release.yml` ran (archives + SHA256SUMS on the Release)
- [ ] Confirm the Homebrew formula updated in kechol/homebrew-tap
- [ ] Confirm `docs.yml` published the docs site
EOF
)"
```

Print the PR URL when done.

## Constraints

- **Never tag.** Tagging triggers `release.yml`; that's the maintainer's
  call.
- **Never push to `main`.** PRs only.
- **Never publish.** GitHub Release + Homebrew formula are done by
  `release.yml` after the tag.
- **No `--no-verify`.** Pre-commit hooks must pass (workflow.md R6).
- **One in-flight release branch at a time** — pre-flight refuses if
  `release/v*` already exists on the remote.

## When NOT to act

- Working tree dirty / branch out of sync / `gh` not authenticated —
  pre-flight catches this; restate the failed check.
- A hotfix that needs to bypass `main` — out of scope; do it manually.
- CI on `main` is red — flag it
  (`gh run list --branch main --limit 1`) and ask before continuing.
  Don't gate the release on CI, but make sure the user sees the color.
